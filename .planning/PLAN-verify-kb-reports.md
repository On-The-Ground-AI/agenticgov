# Plan: Verify KB Reports Against Evidence Quality Grading Frameworks

## Objective

Extend the Causalis quality scoring pipeline so that **every AI-generated report** (evaluation designs, method recommendations, quality scores, ex-ante analyses) is systematically verified against **7 internationally recognized evidence quality grading frameworks** — not just the current ADB + Proposal Quality dual rubric.

This makes the platform's outputs defensible to MDB reviewers, J-PAL affiliates, DFID/FCDO assessors, and education-sector evaluators who each expect compliance with their own standard.

Only 2 of 7 frameworks are operationalized in the scoring API. The other 5 exist as passive KB text (or not at all). No report generated by `/api/evaluate`, `/api/recommend`, `/api/exante`, or `/api/score` is cross-checked against multiple frameworks.

## The 7 Frameworks

| # | Framework | Origin | Focus | Scale |
|---|-----------|--------|-------|-------|
| 1 | **ADB Four-Criterion** | Asian Development Bank IED | MDB project evaluation | 0-3 per criterion |
| 2 | **OECD-DAC** | OECD Development Assistance Committee | Development program evaluation | 6-point (revised 2019) |
| 3 | **J-PAL Evidence Standards** | MIT J-PAL | RCT quality & evidence strength | 4-tier (Strong/Promising/Limited/None) |
| 4 | **Campbell/ROBINS-I** | Cochrane / Campbell Collaboration | Risk of bias in observational studies | Low/Moderate/Serious/Critical per domain |
| 5 | **DFID/FCDO Quality Framework** | UK Foreign, Commonwealth & Development Office | Assurance of evaluation quality | 5-star rating |
| 6 | **WWC Evidence Standards** | US What Works Clearinghouse / IES | Education intervention evidence | 3-tier (Meets/Meets with Reservations/Does Not Meet) |
| 7 | **3ie Quality Assessment** | International Initiative for Impact Evaluation | Systematic review & IE quality | Low/Medium/High risk of bias |

### Phase B: Verification Data Model

**Goal:** Define the schema for storing per-report verification results.

#### Task B1 — Create migration `supabase/migrations/005_report_verification.sql`

```sql
-- Stores verification results for each generated report
CREATE TABLE report_verifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  report_id UUID NOT NULL,              -- FK to evaluation_designs or ad-hoc ID
  report_type TEXT NOT NULL,            -- 'evaluation_design' | 'recommendation' | 'quality_score' | 'exante'
  framework_id TEXT NOT NULL,           -- 'adb' | 'oecd_dac' | 'jpal' | 'campbell' | 'dfid' | 'wwc' | '3ie'
  framework_version TEXT,               -- e.g. '2019' for OECD-DAC revision
  dimensions JSONB NOT NULL,            -- Per-dimension scores
  overall_rating TEXT,                  -- Framework-specific overall rating
  overall_score NUMERIC(4,2),           -- Normalized 0-100
  flags TEXT[],                         -- Critical issues found
  created_at TIMESTAMPTZ DEFAULT now(),
  user_id UUID REFERENCES auth.users(id)
);

-- Summary view: one row per report with all framework scores
CREATE TABLE report_verification_summaries (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  report_id UUID NOT NULL UNIQUE,
  report_type TEXT NOT NULL,
  frameworks_applied TEXT[],            -- Which frameworks were run
  composite_score NUMERIC(4,2),         -- Weighted average across frameworks
  min_score NUMERIC(4,2),               -- Lowest framework score (bottleneck)
  critical_flags TEXT[],                -- Union of all framework flags
  recommendation TEXT,                  -- Overall recommendation
  created_at TIMESTAMPTZ DEFAULT now(),
  user_id UUID REFERENCES auth.users(id)
);

-- RLS: users see their own verifications
ALTER TABLE report_verifications ENABLE ROW LEVEL SECURITY;
ALTER TABLE report_verification_summaries ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users see own verifications"
  ON report_verifications FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users see own summaries"
  ON report_verification_summaries FOR SELECT
  USING (auth.uid() = user_id);
```

### Phase C: Verification API

**Goal:** Create a serverless function that takes a generated report and runs it through all applicable frameworks.

#### Task C1 — Create `api/verify.ts`

New endpoint: `POST /api/verify`

**Request:**
```json
{
  "reportText": "...",
  "reportType": "evaluation_design",
  "frameworks": ["adb", "oecd_dac", "jpal", "campbell"],  // optional subset
  "programContext": { "sector": "education", "method": "rct" }
}
```

**Implementation approach:**

1. **Select applicable frameworks** based on `reportType` and `programContext`:
   - RCT-based designs → J-PAL + ADB + OECD-DAC + 3ie
   - Quasi-experimental → Campbell/ROBINS-I + ADB + OECD-DAC + 3ie
   - Education sector → add WWC
   - DFID-funded → add DFID/FCDO
   - Default: ADB + OECD-DAC + applicable method-specific frameworks

2. **Retrieve framework rubrics** from `knowledge_entries` (category: `evidence_framework`)

3. **Retrieve crosswalk context** from KB via RAG search

4. **Single AI call** with structured prompt:
   - System prompt defines all applicable framework rubrics
   - User message contains the report text + framework context
   - Response is structured JSON with per-framework, per-dimension scores
   - Temperature: 0 for deterministic output

5. **Store results** in `report_verifications` (one row per framework) and `report_verification_summaries`

6. **Return** the full verification summary

**Response:**
```json
{
  "verification": {
    "reportId": "uuid",
    "compositeScore": 78.5,
    "minScore": 62.0,
    "criticalFlags": ["Attrition plan not specified"],
    "recommendation": "Acceptable with revisions",
    "frameworks": [
      {
        "frameworkId": "adb",
        "overallRating": "Successful",
        "overallScore": 82.0,
        "dimensions": { ... }
      },
      {
        "frameworkId": "jpal",
        "overallRating": "Promising",
        "overallScore": 75.0,
        "dimensions": { ... }
      }
    ]
  },
  "sources": [...]
}
```

## Success Criteria

1. Every report generated by the platform can be verified against 2+ applicable frameworks
2. Verification results are stored and auditable
3. Users see a clear composite score with framework-specific breakdowns
4. Critical flags (e.g., "no attrition plan", "no pre-analysis plan") surface automatically
5. Admin can see aggregate verification quality across all reports
6. Verification adds no more than 3 seconds to existing API response times (when enabled)
7. Feature flag allows instant disable if issues arise

---

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| AI hallucinating framework scores | Ground all scoring in KB-retrieved rubric text; temperature=0; structured JSON output |
| Latency increase from multi-framework verification | Single AI call with all frameworks in one prompt; async verification option |
| Framework definitions become outdated | Version field in `report_verifications`; KB documents are easily updateable |
| Users confused by multiple scores | Composite score as primary display; framework detail is opt-in expandable |
| Token limits exceeded with 7 frameworks | Limit to applicable frameworks per report type (typically 3-4, not all 7) |
